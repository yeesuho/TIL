# IoC, DI 뭘까?

2025.02.22

### DI (Depnendency Injection, 의존성 주입)
객체가 필요한 의존성을 직접 생성하지 않고, 외부에서 주입 받는 개념

<br>

### IoC (Inversion of Control, 제어의 역전)
객체의 생성과 의존성 관리를 개발자가 아니라 프레임워크가 해주는 개념
<br>

#### 그래서 IoC 컨테이너란:
 IoC 자체는 개념일 뿐 이를 구현하는 방식이 필요함 <br>
 ->IoC를 적용하려면 객체의 생성, 관리, 의존성 주입을 해줄 무언가가 필요
 <br>IoC 개념을 실제로 구현하는 도구가 바로 IoC 컨테이너
 
 #### IoC 컨테이너 역할:
⚬ 객체를 자동으로 생성<br>
⚬ 객체 간의 의존성을 자동으로 주입(DI 적용)<br>
⚬ 객체의 생명 주기 관리
 
<br>

## IoC와 DI 관계 (비유)
ex) 자동차 공장(**프레임워크**)가 자동차(**객체**)를 만드는 과정
<br>
**IoC**: 공장이 자동차 생산을 직접 관리 (**객체의 생성 및 흐름을 컨트롤**)
<br>
**DI**: 공장이 자동차 부품(**의존성**)을 직접 조립해서 넣어줌(**의존성 주입**)

<br>

## IoC 컨테이너 없이 DI가 존재 가능할까?
2026.04.27

IoC 컨테이너 없이도 DI는 적용할 수 있습니다<br>

예를 들어 개발자가 직접 객체를 생성한 뒤 생성자나 메서드를 통해 필요한 의존성을 전달하면 DI가 됩니다<br>

하지만 IoC 컨테이너를 사용하면 객체 생성, 생명주기 관리, 의존성 연결을 프레임워크가 자동으로 처리해주기 때문에 DI를 훨씬 편리하고 강력하게 사용할 수 있습니다<br>

즉, IoC 컨테이너 없이 DI는 가능하지만, 이 경우 의존성 생성과 연결을 개발자가 직접 관리해야 합니다<br>

반대로 DI는 IoC를 구현하는 대표적인 방법 중 하나일 뿐이므로, DI 없이도 IoC는 존재할 수 있습니다
